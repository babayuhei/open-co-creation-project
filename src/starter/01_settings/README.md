# AI小説執筆SPMモデル

> **ライセンス:** このディレクトリ内のすべてのファイルは [MIT License](https://github.com/babayuhei/confession-of-a-loser/blob/main/LICENSE-MIT) で公開されています。自由に改変・再利用できます（帰属表示を維持してください）。

---

SPMモデルは、このプロジェクトで構築したAI長編小説執筆のフレームワークです。3つの層を定義し、情報の解像度を段階的に上げながら原稿を生成します。

```
S: Settings（設計図）   → 不変の定義。物語の憲法。Single Source of Truth。
         ↓
P: Plots（中間生成物）  → Sを具体化する中間言語。シーン単位のロジック。
         ↓
M: Manuscripts（原稿）  → 読者が目にする最終アウトプット。
```

推敲フェーズでは逆方向にも伝播します——M層で矛盾が出たらP層へ遡り、さらにS層まで修正してから再生成する、というバックプロパゲーションを繰り返しました。

---

## S: Settings（設計図）

**役割：** 物語全体の「Single Source of Truth」。AIに世界観を理解させるための不変定義です。ここに書かれた設計が変わらない限り、何万字の原稿を生成しても一貫性が保たれます。ソフトウェア開発でいえばMVCの **Model（Schema/DB）** に相当します。

| ファイル | 内容 |
|:--------|:-----|
| [storyline.md](storyline.md) | 作品コンセプト・核心メッセージ・全体構成・各部テーマ |
| [os_theory.md](os_theory.md) | OS理論の体系（自分本位↔他者本位、思考型↔行動型の4象限）|
| [characters.md](characters.md) | 登場人物のOS・形成背景・変容プロセス・口調設計 |
| [style_guide.md](style_guide.md) | 文体ルール・NG表現・シーンタイプ別構成パターン |
| [app_design.md](app_design.md) | PDCA・GROWモデルの埋め込み設計・[L]シーン8ステップ構造 |

**読み順：**
```
storyline.md → os_theory.md → characters.md → style_guide.md → app_design.md
```

**設計の核心：** `storyline.md` をS層に置く決断が最大の肝です。ストーリーラインを「静的な仕様」として固定することで、物語の背骨がAIに振り回されるのを防いでいます。

---

## P: Plots（中間生成物）

**役割：** Sという抽象的な設計図を、Mという具体的な描写に変換するための**中間言語**です。シーン単位で「何が起きるか」を定義することで、いきなり原稿を書かせるよりも論理的な破綻が激減します。MVCでいえば **Controller（Logic）** に相当します。

| ファイル | 内容 |
|:--------|:-----|
| [全体構成概観](../story/cross_part_overview.md) | 全体の時系列・伏線マップ・部ごとのテーマ |
| [第1部サマリー](../story/scene_summaries_part1.md) | 第1部・全シーンのビート・PDCA層・伏線情報 |
| [第2部サマリー](../story/scene_summaries_part2.md) | 第2部・全シーンのビート・PDCA層・伏線情報 |
| [第3部サマリー](../story/scene_summaries_part3.md) | 第3部・全シーンのビート・PDCA層・伏線情報 |

各シーンのサマリーには「ビート（転換点）」と「PDCA層（学習フレームワーク上の位置）」を必ず記載しています。この地図情報がないと、AIは毎シーン独立した起承転結を書いてしまいます。

---

## M: Manuscripts（原稿）

**役割：** S・P 2層の結果が反映される最終アウトプット。読者が目にする小説テキストそのものです。MVCでいえば **View（UI）** に相当します。

本プロジェクトでは **1シーン1ファイル**で管理しています（例：`03_manuscripts/01_01_四月の営業フロア.md`）。S層（設定ファイル）はMIT Licenseで公開していますが、M層（原稿）は小説本体として公開しています。

→ **小説本体はこちら：** [失敗者の告白](https://babayuhei.github.io/confession-of-a-loser-workspace/)

---

## 推敲フロー（バックプロパゲーション）

```
M層で矛盾・品質問題を発見
  ↓
P層（シーンサマリー）を修正
  ↓
S層（設定）まで影響するか判断
  ↓
S層修正 → P層再生成 → M層再生成
```

この「上位レイヤーへの遡り → 下位への伝播」をソフトウェア開発のリファクタリングと同じように運用し、23バージョンにわたって繰り返しました。
